# Module Developer Guide

Every feature in Vexaltrix is a **module** — a self-contained unit that can be enabled or disabled at runtime via the admin UI. Modules are loaded **only when enabled**.

---

## File Structure

Each module lives entirely inside `modules/{slug}/`. PHP and JS coexist in the same folder.

```
modules/{slug}/
├── {Name}Module.php           ← PHP: registers WP hooks, blocks, REST routes
├── block.json                 ← Block metadata (if the module registers a block)
├── src/
│   ├── index.js               ← Admin settings card + block editor script (React)
│   ├── frontend.js            ← Public-facing JS (optional)
│   └── components/
│       ├── Edit.js            ← Gutenberg block Edit component
│       ├── {Name}Settings.js  ← Admin settings card component
│       └── ...
└── languages/                 ← Module-specific .po / .mo files
```

**Compiled output** (generated by `pnpm plugin:build`, git-ignored):

```
assets/build/modules/{slug}/
├── index.js        ← admin card + block editor
├── index.css       ← editor styles
├── frontend.js     ← public JS
└── style-index.css ← public styles
```

## Step 1 — Create the PHP class

```php
// modules/my-feature/MyFeatureModule.php
namespace Vexaltrix\Modules\MyFeature;

use Vexaltrix\Core\Module\AbstractModule;

class MyFeatureModule extends AbstractModule {

    public function slug(): string        { return 'my-feature'; }
    public function label(): string       { return __( 'My Feature', 'vexaltrix' ); }
    public function description(): string { return __( 'Does something useful.', 'vexaltrix' ); }
    public function dependencies(): array { return []; } // or e.g. ['example']

    public function boot(): void {
        // Called ONLY when this module is enabled.
        add_action( 'init',          [ $this, 'registerBlock'  ] );
        add_action( 'rest_api_init', [ $this, 'registerRoutes' ] );

        // Fire action so third-party code can extend this module.
        $this->doAction( 'booted' );
    }

    public function registerBlock(): void {
        $blockJson = VEXALTRIX_PATH . 'modules/my-feature/block.json';
        if ( file_exists( $blockJson ) ) {
            register_block_type( $blockJson, [ 'render_callback' => [ $this, 'renderBlock' ] ] );
        }
    }

    public function renderBlock( array $attributes, string $content ): string {
        ob_start();
        ?>
        <div <?php echo get_block_wrapper_attributes(); ?>>
            <p><?php echo esc_html( $attributes['label'] ?? '' ); ?></p>
        </div>
        <?php
        return (string) ob_get_clean();
    }

    public function registerRoutes(): void {
        register_rest_route( 'vexaltrix/v1', '/' . $this->restBase(), [
            'methods'             => \WP_REST_Server::READABLE,
            'callback'            => fn() => rest_ensure_response( [ 'active' => true ] ),
            'permission_callback' => '__return_true',
        ] );
    }
}
```

## Step 2 — Register in composer.json

Add the module PHP file to `classmap` in `composer.json` so Composer can autoload it:

```json
"autoload": {
    "psr-4": {
        "Vexaltrix\\": "app/",
        "Vexaltrix\\Modules\\": "modules/"
    },
    "classmap": [
        "modules/example/ExampleModule.php",
        "modules/contact-form/ContactFormModule.php",
        "modules/analytics/AnalyticsModule.php",
        "modules/my-feature/MyFeatureModule.php"
    ]
}
```

Then re-generate the autoloader:

```bash
composer dump-autoload -o
# Hoặc khi mới install:
composer install
```

## Step 3 — Register in Plugin.php

Add the class-string to `Plugin::$modules`:

```php
// app/Plugin.php
private array $modules = [
    \Vexaltrix\Modules\Example\ExampleModule::class,
    \Vexaltrix\Modules\ContactForm\ContactFormModule::class,
    \Vexaltrix\Modules\Analytics\AnalyticsModule::class,
    \Vexaltrix\Modules\MyFeature\MyFeatureModule::class,  // ← add here
];
```

The module now appears automatically in the **Modules** admin UI.

## Step 4 — Admin settings card (optional)

Create `modules/my-feature/src/index.js`. It mounts into a slot that `ModuleCard` creates when the module is enabled:

```js
// modules/my-feature/src/index.js
import { render } from '@wordpress/element';
import MySettings from './components/MySettings';

const mount = () => {
    const root = document.getElementById( 'vx-module-settings-my-feature' );
    if ( root ) render( <MySettings />, root );
};

document.readyState === 'loading'
    ? document.addEventListener( 'DOMContentLoaded', mount )
    : mount();
```

Webpack **auto-discovers** `modules/*/src/index.js` — no config changes needed.

## Step 5 — Block metadata (if registering a block)

```json
// modules/my-feature/block.json
{
    "$schema": "https://schemas.wp.org/trunk/block.json",
    "apiVersion": 3,
    "name": "vexaltrix/my-feature",
    "title": "My Feature",
    "textdomain": "vexaltrix",
    "editorScript": "file:../../assets/build/modules/my-feature/index.js",
    "style":        "file:../../assets/build/modules/my-feature/style-index.css"
}
```

The `file:../../` paths are relative to `block.json` and resolve to `assets/build/modules/my-feature/`.

## AbstractModule — Helper Reference

| Method | Returns | Description |
|--------|---------|-------------|
| `$this->handle('suffix')` | `string` | Asset handle: `vexaltrix-module-{slug}-suffix` |
| `$this->restBase()` | `string` | REST sub-path: `modules/{slug}` |
| `$this->doAction('event')` | `void` | Fires `vexaltrix/module/{slug}/event` |
| `$this->applyFilter('hook', $val)` | `mixed` | Applies `vexaltrix/module/{slug}/hook` |
| `$this->services->get(Class::class)` | `object` | Resolve shared service from DI container |

---

## ModuleRegistry — WP Hooks

| Hook | Description |
|------|-------------|
| `vexaltrix/module/registered` | Fired after each module is registered |
| `vexaltrix/module/{slug}/booted` | Fired after `boot()` runs |
| `vexaltrix/module/{slug}/enabled` | Fired after enabling via REST API |
| `vexaltrix/module/{slug}/disabled` | Fired after disabling via REST API |
| `vexaltrix/modules/states` | Filter — override enabled states before boot |
| `vexaltrix/modules/booted` | Fired after all modules boot |

---

## REST API — Module Management

| Method | Endpoint | Auth | Description |
|--------|----------|------|-------------|
| `GET` | `/wp-json/vexaltrix/v1/modules` | `manage_options` | List all modules with status |
| `POST` | `/wp-json/vexaltrix/v1/modules/{slug}/enable` | `manage_options` | Enable a module |
| `POST` | `/wp-json/vexaltrix/v1/modules/{slug}/disable` | `manage_options` | Disable a module |

## Dependency Management

```php
public function dependencies(): array {
    return [ 'contact-form' ]; // slug(s) this module requires
}
```

The registry enforces:

- **Enable blocked** if any dependency is disabled
- **Disable blocked** if any dependent module is still enabled
- **Boot order** — topological sort ensures deps always boot first

## i18n

PHP strings are extracted into the main `languages/vexaltrix.pot`.  
Module-specific JS translations go in `modules/{slug}/languages/`.

```bash
pnpm plugin:i18n   # regenerate .pot + compile .po → .mo + JS JSON
```
